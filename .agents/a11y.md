# Accessibility Guidelines

This project uses [React Aria Components](https://react-spectrum.adobe.com/react-aria/components.html) as its accessibility foundation. React Aria provides WAI-ARIA-compliant keyboard interactions, focus management, and ARIA semantics out of the box. These guidelines cover what to be aware of when building on top of that foundation.

## Core Principles

1. **Lean on React Aria.** Don't manually add ARIA roles, states, or keyboard handlers that React Aria already provides. Adding redundant attributes can conflict with the library's internals.
2. **Every interactive element needs an accessible name.** If there's no visible text label, provide an `aria-label` or `aria-labelledby`.
3. **Every form field needs a visible label.** The toolkit's form field components enforce `label` as a required prop -- do not circumvent this.
4. **Decorative elements must be hidden.** Use `aria-hidden="true"` on icons, SVGs, and other visual-only content that doesn't convey meaning.
5. **Use semantic HTML.** Prefer `<button>`, `<a>`, `<nav>`, `<main>`, `<header>`, `<fieldset>`, etc. over generic `<div>` and `<span>` elements.

## Labeling

### Buttons and Icon Buttons

- `Button` with visible text is self-labeling.
- `IconButton` requires `aria-label` (enforced by TypeScript -- it's a required prop).
- If a button's visible text is ambiguous out of context (e.g. "Edit"), consider adding `aria-label` with more detail (e.g. `aria-label="Edit recipe"`).

### Form Fields

- All form field components (`FormTextField`, `FormTextareaField`, `FormSelectField`, etc.) require a `label` prop. This renders a visible `<Label>` that React Aria automatically associates with the input.
- Use the `<Description>` component for supplementary help text. React Aria links it to the input via `aria-describedby`.
- `<FieldError>` is automatically linked to the input via `aria-describedby` when validation fails. React Aria also sets `aria-invalid="true"`.

### Compound Fields

For compound fields that don't map to a single React Aria field context (e.g. `InputGroupSelectField`), you need manual ARIA wiring:

- Use `useId()` from React to generate stable IDs.
- Apply `aria-labelledby` to the group element, pointing to the visible label's ID.
- Apply `aria-describedby` to link description and error elements.

### Grouped Content

- Use `<fieldset>` + `<legend>` for groups of related form fields. Use the `sr-only` class on the `<legend>` if there's already a visible heading, and mark the visible heading with `aria-hidden="true"` to prevent double announcement.
- Use `aria-labelledby` on `Alert` components when they have a title element.

## Keyboard and Focus

### Focus Rings

The toolkit provides two reusable focus ring systems:

- **`buttonFocusRing`** -- used on buttons, icon buttons, links, tabs, and pagination controls. Shows a ring on `:focus-visible` only (keyboard navigation, not mouse clicks).
- **`inputFocusRing`** -- used on inputs with three variants:
  - `focused` (default) -- ring on any focus (`data-[focused]`)
  - `focusVisible` -- ring on keyboard focus only (`data-[focus-visible]`), used by selects and checkboxes
  - `focusWithin` -- ring when any child is focused (`data-[focus-within]`), used by `InputGroup`

All input focus ring variants include destructive (error) state styling.

When building new interactive components, use one of these existing systems rather than writing custom focus styles.

### Keyboard Navigation

React Aria handles keyboard navigation patterns automatically:

| Component | Key Interactions |
|-----------|-----------------|
| Button | `Space` / `Enter` to activate |
| Checkbox | `Space` to toggle |
| Select | `Space` / `Enter` / `ArrowDown` to open; arrows to navigate; `Escape` to close; type-ahead |
| Menu | Same as Select, plus `ArrowRight` / `ArrowLeft` for submenus |
| Tabs | `ArrowLeft` / `ArrowRight` (horizontal); `ArrowUp` / `ArrowDown` (vertical); `Home` / `End` |
| Listbox | Arrows to navigate; `Space` / `Enter` to select; type-ahead |
| Popover | `Escape` to close; focus is trapped within |

Do not override these patterns. If a custom component needs keyboard interaction, follow the [WAI-ARIA Authoring Practices](https://www.w3.org/WAI/ARIA/apg/patterns/).

### Disabled States

- Use `isDisabled` on React Aria components (not the native `disabled` attribute directly).
- The toolkit's `inputDisabled` variant applies consistent disabled styling (`bg-slate-50`, `opacity-50`).
- Buttons use `disabled:pointer-events-none disabled:opacity-50`.
- The `FormSubmitButton` automatically disables when `canSubmit` is false.

## Screen Readers

### Hiding Decorative Content

- Use `aria-hidden="true"` on icons, SVGs, decorative images, and visual-only elements.
- Use `role="presentation"` or `role="none"` on wrapper elements that exist only for layout (e.g. `Badge`, `InputGroupAddon`).
- Pagination ellipsis (`...`) should be `aria-hidden="true"`.

### Providing Text Alternatives

- Use the `textValue` prop on `ListboxItem` and `MenuItem` when children are not plain strings (e.g. items with icons). The toolkit auto-derives this from string children, but complex children need it explicitly.
- `Avatar` falls back to `alt={name || "Avatar"}` -- always provide a meaningful `name` or `alt` prop.

### Live Regions

- The `Alert` component uses `role="alert"` (implicit `aria-live="assertive"`). Content rendered inside an Alert is announced immediately by screen readers.
- For less urgent announcements (loading states, status updates), use `aria-live="polite"` on the container element.

### Navigation and Page Structure

- Use proper landmark elements: `<header>`, `<nav>`, `<main>`, `<footer>`.
- Set `aria-current="page"` on the active navigation link (the toolkit's `Link` component supports this).
- Set `aria-current="page"` on the active pagination button.
- The root `<html>` element must have a `lang` attribute (currently `lang="en"`).

## Styling Accessibility States

The `tailwindcss-react-aria-components` plugin maps React Aria's data attributes to Tailwind variants:

| Data Attribute | Tailwind Variant | Used For |
|----------------|-----------------|----------|
| `data-focused` | `focused:` | Element has focus |
| `data-focus-visible` | `focus-visible:` | Keyboard focus only |
| `data-selected` | `selected:` | Selected state |
| `data-pressed` | `pressed:` | Active/pressed state |
| `data-disabled` | `disabled:` | Disabled state |
| `data-invalid` | `invalid:` | Validation error state |
| `data-focus-within` | `focus-within:` | Child has focus |

Use these variants to style interactive states. They ensure visual feedback aligns with the underlying ARIA state.

## Linting

- The web app (`apps/web/`) has Biome's recommended `a11y` rules enabled.
- The Storybook `@storybook/addon-a11y` addon runs [axe-core](https://github.com/dequelabs/axe-core) audits on every story in the toolkit. Check the "Accessibility" panel in Storybook for violations.

## Checklist for New Components

When adding a new component, verify:

- [ ] Interactive elements have an accessible name (visible text, `aria-label`, or `aria-labelledby`)
- [ ] Decorative elements are hidden with `aria-hidden="true"`
- [ ] Focus is visible and uses the appropriate ring system (`buttonFocusRing` or `inputFocusRing`)
- [ ] Keyboard navigation works without a mouse
- [ ] Disabled states use `isDisabled` and appropriate styling
- [ ] Color is not the only means of conveying information (pair with icons, text, or patterns)
- [ ] A Storybook story exists and the a11y addon panel shows no violations
